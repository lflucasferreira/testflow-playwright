What is Playwright?

Playwright is a Microsoft automation framework for end-to-end and API testing — with native support for Chromium, Firefox, and WebKit.

  • Written in TypeScript / JavaScript
  • Modern API with auto-wait — less flakiness
  • Fixtures page, request, context per test
  • UI Mode, trace viewer, screenshots, and HTML report
  • Native parallelism and isolation between workers
Unlike Cypress, Playwright uses WebDriver BiDi / native browser protocols and runs outside the application context — supporting multiple browsers and API-only tests in the same runner.

Why use Playwright?

Speed & parallelism

Parallel workers and isolated browser contexts per test.

Resilient locators

getByTestId, role, text — recommended by official docs.

Rich debugging

UI Mode, headed, Inspector, and trace on retry.

Unified E2E + API

page for UI and request for REST contracts.

beforeEach vs beforeAll

// beforeEach — fresh state per test
test.beforeEach(async ({ page, request }) => {
  await loginViaApi(page, request)
  await new DashboardPage(page).shouldBeLoaded()
})
// beforeAll — shared API response
let res: APIResponse
let body: { users: User[] }

test.beforeAll(async ({ request }) => {
  const start = Date.now()
  res = await request.get('/api/users')
  duration = Date.now() - start
  body = await res.json()
})

test('returns status 200', () => {
  expect(res.status()).toBe(200)
})

beforeAll avoids N identical calls; beforeEach isolates UI state.

test.step — readable report

test('GET /resources — Authorization header omitted', async ({ playwright }, testInfo) => {
  await test.step('New context without Authorization header', async () => {
    const ctx = await playwright.request.newContext({ baseURL })
    try {
      const response = await ctx.get('/api/resources')
      await attachHttpExchangeReport(testInfo, {
        label: 'resources-no-auth',
        method: 'GET',
        url: '/api/resources',
        response,
      })
      expect(response.status()).toBe(403)

      await ctx.dispose()

export function validateSchema(obj, schema: Schema) {
  for (const [key, type] of Object.entries(schema)) {
    expect(obj).toHaveProperty(key)
    expect(typeof obj[key]).toBe(type)
  }
}

Auth helpers — fast setup

async function injectAuth(page, session) {
  await page.addInitScript((auth) => {
    sessionStorage.setItem('sandbox-auth', JSON.stringify(auth))
    sessionStorage.setItem('sandbox-token', auth.token)
  }, session)
}

Data-driven — runPatchTests()

export function runPatchTests(context, { payloads, tag }) {
  for (const [label, payload] of Object.entries(payloads)) {
    test.describe(label, () => {
      test(`204 — PATCH accepted, GET profile matches`, async ({ request, authToken }) => {
        const patchRes = await request.patch(PATCH_PATH, { headers, data: payload })
        expect(patchRes.status()).toBe(204)
        const profile = await pollGetUntilMatch(request, GET_PATH, headers, payload)
        expect(profile).toMatchObject(normalizePayload(payload))
      })
    })
  }
}

PATCH → poll GET — eventual consistency

async function pollGetUntilMatch(request, path, headers, expected, deadlineMs = 10_000) {
  const start = Date.now()
  while (Date.now() - start < deadlineMs) {
    const res = await request.get(path, { headers })
    if (res.status() === 200) {
      const body = normalizeProfile(await res.json())
      if (matchesExpected(body, expected)) return body
    }
    await new Promise((r) => setTimeout(r, 500))
  }
  throw new Error('GET did not match PATCH payload within deadline')
}

Anatomy — LoginPage.ts

export class LoginPage {
  constructor(private readonly page: Page) {}

  emailInput() { return this.page.getByTestId('login-email') }
  submitBtn()  { return this.page.getByTestId('login-submit') }

  async visit() {
    await this.page.goto('/web/login.html')
    return this
  }

  async loginWith(email: string, password: string) {
    await this.emailInput().fill(email)
    await this.passwordInput().fill(password)
    await this.submitBtn().click()
    return this
  }

  async shouldRedirectToDashboard() {
    await expect(this.page.getByTestId('page-dashboard')).toBeVisible()
    await expect(this.page).toHaveURL(/\/web\/dashboard\.html/)
  }
}

Accessibility — axe-core

a11y suite — scans with @axe-core/playwright on key pages. Doc: a11y.md

import AxeBuilder from '@axe-core/playwright'

async function assertNoCriticalViolations(page, scope?: string) {
  let builder = new AxeBuilder({ page }).disableRules(['color-contrast'])
  if (scope) builder = builder.include(scope)
  const results = await builder.analyze()
  const critical = results.violations.filter((v) => v.impact === 'critical')
  expect(critical).toEqual([])
}

Best practices

  • Page Objects in pages/ — isolated locators, actions, and assertions
  • getByTestId + data-testid — same convention as testflow-cypress
  • loginViaApi in beforeEach — feature specs without repeating UI login
  • beforeAll + shared response — API contracts without N identical requests
  • validateSchema — declarative REST response types
  • page.route + waitForRequest — spy payload and mock errors
  • JSON fixtures — deterministic data (credentials, team-member)
  • One scenario per test() — easy-to-diagnose failures in the report
  • projects in config — 18 suites; CI runs all with npm test
  • test.step + attachHttpExchangeReport — API debug in HTML report
  • EXPECT constants — named status codes, not magic numbers
  • Tags @smoke / @regression — filter local runs with test:grep:*
  • globalSetup + token cache — heavy setup once per run
  • visitAuthenticated — navigate authenticated to any route
  • AxeBuilder + @a11y — accessibility scans without critical violations
  • toHaveScreenshot + @visual — baselines with maxDiffPixelRatio: 0.05
  • ShellPage.navItems() — data-driven navigation in the app shell
  • Factories in support/factories/ — reusable wizard and PATCH payloads
  • readFixture + expectSameMembers — golden contracts without JSON Schema
  • JsonPatchBuilder + runPatchTests() — enterprise data-driven PATCH
  • Read the block-by-block walkthrough in docs/en/ or docs/pt/
  • npm run test:ui in dev — immediate visual feedback
